# 如何给比赛机器人做一个可调试的定位系统：R1_LocaterV2 的一次工程复盘

> 项目地址：https://github.com/lwbscu/R1_LocaterV2  
> 关键词：机器人定位、STM32G4、传感器融合、上位机、real2sim

![R1_LocaterV2](./r1-locaterv2-poster.png)

这次我们做的是 R1 机器人的新定位板。它不是一个“传感器读数展示板”，而是一套围绕实车调试建立起来的定位闭环：车上固件负责稳定采集和输出，Windows 上位机负责实时地图、日志、回放和仿真验证。

## 这个问题为什么难

比赛机器人定位看起来可以直接上 Lidar，但实际调车时会遇到几个很具体的问题。

第一，Lidar 很准，但它给出的坐标语义必须和底盘控制一致。我们现在的约定是：机器人在红方左上启动区几何中心上电时，`x=0, y=0, yaw=0`。地图正东是 `+X`，正北是 `+Y`。这比“场地中心是 0 点”更符合底盘使用方式。

第二，编码轮灵敏，但会积累误差。它适合做两帧 Lidar 之间的高频插值，不适合作为长时间绝对定位。

第三，H30 的 yaw 稳定，可以作为 DT35 射线和编码轮坐标变换的姿态基准。这里不需要再在定位板上做复杂 Kalman 滤波，成品惯导已经提供了可用的 yaw。

第四，DT35 的数值本身不难读，难的是判断它测到的是墙、梅林、坡道、人、队友机器人，还是场地上根本没有摆的挡板。

## 我们最终采用的结构

R1_LocaterV2 的传感器配置是：

- H30 MINI：提供 yaw。
- 双正交编码轮：提供高频位移。
- Lidar：提供启动坐标系下的相对绝对位姿。
- 双 DT35：左右侧向测距，用于墙体距离约束。

PC 上位机使用 PySide6 写，不用 ROS、不用 Web，也不依赖 Foxglove/RViz。它接收 STM32 USART1 通过无线串口发来的 CSV 数据，并实时显示地图、机器人、轨迹、DT35 射线和传感器状态。

![上位机](./r1-locaterv2-ui-demo.png)

## 为什么上位机很重要

调定位算法时，只看串口数字很容易误判。比如 DT35 显示 5000 mm，你必须知道它这束光在当前 yaw 下应该打到哪面墙；如果穿过了梅林，或者打到了长杆架空隙，数值就不能直接用于修正。

上位机解决的就是这个问题：

- 地图按真实尺寸显示，单位是 cm。
- 机器人底盘贴图固定为 83 cm x 83 cm，中心对齐定位输出。
- DT35 根据安装 offset 和 H30 yaw 画射线。
- 场地墙体模型区分可用墙、强干扰区和特殊障碍。
- 每次采集都会保存 raw 串口、结构化 CSV 和地图截图。

这样我们可以在实车跑完后离线回放，逐帧看“当时车在哪里、DT35 打到哪里、算法为什么信或不信这条距离”。

## real2sim：用仿真反过来检查实车

这个项目里我最重视的是 real2sim。它不是做一个漂亮仿真图，而是把实际传感器数据放回理想场地图里验证。

流程是：

1. Lidar 给出启动局部坐标下的参考位置。
2. H30 yaw 决定车体朝向。
3. 编码轮提供相邻 Lidar 帧之间的位移插值。
4. DT35 射线和地图墙体求交，判断距离是否可信。
5. 如果可信，再作为位置约束；如果命中长杆架、边角、缺失墙体或不规则区域，就降低权重或丢弃。

这解决了一个实际问题：练习场往往不完全等于比赛场。有些外墙没摆，有些地方地面不平，DT35 偶尔会打到地面或被人挡住。如果算法把所有 DT35 数值都当真，定位会被错误观测带偏。

## 当前结果

当前工程已经跑通：

- STM32G4 固件。
- USART1 轻量定位 CSV 调试输出。
- USART2 底盘主控定位帧。
- H30 / 编码轮 / Lidar / DT35 数据链路。
- PySide6 实时地图、采集、回放、仿真分析。
- 剪映宣传视频草稿和 GitHub README 素材。

初步实车跑完整路径时，yaw 累计误差约 `0.04 deg`。离线仿真基准中，代表路径平均 XY RMS 从约 `5.64 cm` 降到约 `1.97 cm`。这些数字还需要在正式比赛场地继续复核，但工程链路已经具备持续迭代能力。

## 经验总结

这次最大的收获是：定位系统的核心不是某一个滤波公式，而是让所有传感器、坐标系、地图、日志和可视化说同一种语言。

如果没有上位机，就很难发现贴图偏转、坐标原点选错、DT35 穿墙、墙体模型错位这种问题。如果没有日志回放，每次改算法都要重新推车，效率很低。如果没有场地模型，DT35 的距离只能是一个数字，无法判断它该不该被相信。

所以 R1_LocaterV2 的重点不是“最终算法一次写对”，而是把调试系统做成可以持续逼近正确的结构。
